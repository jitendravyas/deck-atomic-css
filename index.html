What problems does this solve?

Overhead: building and maintaining good CSS is really hard and time-consuming

Communication: it can be hard for designers and developers to speak a common language, and hard for anyone to understand unfamiliar CSS

Incompleteness: new product feature ‚û°Ô∏è additional UI components or variants needed ‚û°Ô∏è more CSS added ‚û°Ô∏è repeat üò¢

Complexity: the best CSS is the least CSS possible

Benefits

Create new UI components with no new CSS (!!!)

the stylesheet starts small and stays that way

no more writing / rewriting / editing CSS again

atomic classes can be mixed and matched to create a highly expressive array of new UI components without adding to or even opening a stylesheet

Reusable

Content classes are specific: they are necessarily scoped to the specific content element they describe

Presentational classes are universal: they can be used wherever the visual need exists

Encapsulation is your best friend

Stop worrying about specificity: flat list of classes means no more worrying about global scope, specificity wars, overrides, descendant selectors

Easily refactorable: by encapsulating work in an individual block of html, it‚Äôs easy to refactor something with the confidence you aren't breaking anything else

Use components anywhere, in any context: your UI is now 100% portable

Beginner-friendly

Democratize design: a small, fixed, descriptive system of classes simplifies the conceptual overhead, lessens the learning curve, and can democratize the design process üá∫üá∏

Common questions

Doesn‚Äôt this makes the DOM ugly?

it definitely means that elements can and will have multiple classes

stylistically, that can be ugly at first blush, but it's a worthwhile tradeoff

What if I have to change something on a hundred different pages?

If this worries you, I'm more worried about your templating architecture.

If the markup for something is defined multiple times across templates, it should be consolidated to a single place.

How is this better than inline styles?

inline styles don't support media queries or pseudo elements

inline styles are 1:1 for element styles; a class is 1 to many

atomic css can leverage the benefits of framework level variables

atomic classes are faster to type and easier to read

Is this semantic?

yes!

classes don't affect HTML semantics, and please please please use well structured, accessible, semantic HTML.

a great resource: About HTML Semantics and Front-End Architecture, by @necolas
